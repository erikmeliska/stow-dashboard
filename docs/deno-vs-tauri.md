# Deno Desktop vs Tauri — comparison

Companion to `docs/superpowers/specs/2026-07-04-deno-desktop-shell-design.md`.
Both shells wrap the same Next.js 16 standalone app. Tauri: port 3088. Deno: port 3087.

## Node API compatibility notes

- **Critical, load-bearing finding:** `deno desktop .` (and the compiled `.app`) run under Deno's permission system same as `deno run`/`deno compile`. Without explicit permission flags, the very first `require`d Next.js internal module that reads `process.env` throws and kills the process before the server can bind to a port:

  ```
  [desktop] Deno runtime error: NotCapable: Requires env access to "__NEXT_PRIVATE_CPU_PROFILE", specify the required permissions during compilation using `deno compile --allow-env`
      at Object.getEnv [as get] (ext:deno_os/30_os.js:1:1634)
      ...
      at Object.<anonymous> (.../node_modules/next/dist/server/lib/cpu-profile.js:2:43)
  ```

  Fix: run/build with `-A` (allow-all). This app already needs unrestricted env/read/write/net/run/sys access to do its job (scan the filesystem, spawn `scc`/`git`/`lsof`/`docker`/`osascript`, write JSONL) — identical to what it gets for free under Tauri + system Node, where there is no permission sandbox at all. **Tasks 2–4's `deno:run`/`deno:build` npm scripts and the `deno run -A src-deno/server.ts` verification step must include `-A`** (the plan's Task 2/Task 3 command blocks did not originally show this flag explicitly for `deno:run`/`deno:build` — see the plan-file diff below).
- `next start` printed a warning because the project's `next.config.mjs` sets `output: 'standalone'`: `"next start" does not work with "output: standalone" configuration. Use "node .next/standalone/server.js" instead.` It still started and served correctly in this spike (auto-detect runs the ordinary `next-start.js`, not our standalone `server.js`), but this is the exact reason Task 3 builds a custom `server.ts` that imports the **standalone** `server.js` directly instead of relying on auto-detect's `next start` path — auto-detect is fine for the spike gate, but the real shell should use the custom entrypoint to avoid depending on the non-standalone Next server and to get the writable-cwd data-file handling.
- Auto-detect's VFS embedding is scoped to the framework's own build output directory (`.next/` for Next.js) as documented in "Working directory & assets" — it will not pick up sibling directories like `data/` or dotfiles like `.env.local` that our app also needs at `process.cwd()`. Confirmed by direct inspection of the self-extracted VFS directory contents (only `.deno_desktop_entry-*.ts`, `.next/`, `node_modules/`, `package.json`, `public/`, `src-tauri/` were present — no `data/`, no `.env.local`). This validates the plan's Task 3 design of syncing writable state into `~/Library/Application Support/StowDashboardDeno` explicitly rather than relying on auto-detect.
- Auto-detect embeds far more than necessary when pointed at `.` with no exclusions: it picked up all of `node_modules/` (433MB), the full `.next/` build (610MB), and even `src-tauri/target/release/**` (100MB, including a stale `Stow Dashboard.app` bundle and its `standalone/node_modules`) because nothing under the project root was excluded from the scan. Total embedded: 973.54MB, producing a ~1GB `.app`. Tasks 2–4's custom entrypoint + curated `src-deno/standalone/` + `--include` flags avoid this bloat entirely — worth calling out as a concrete DX/size advantage of the custom-entrypoint approach over auto-detect for this specific repo (which has `src-tauri/target` sitting in the same tree).
- No crashes, hangs, or Node-API-missing errors were observed anywhere else in the exercised surface (child_process spawn/exec for git, lsof, scc, npm scripts; SSE streaming responses; fs read/write) — Deno's Node compat handled all of it once permissions were granted.
- Docs verification (Step 2) found all API signatures in the plan's Task 2–4 code blocks match the live docs exactly as written: `Deno.BrowserWindow({title,width,height})`, `win.navigate()`, `win.show/hide/focus/isVisible()`, `close` event + `e.preventDefault()`, `new Deno.Tray()`, `tray.setIcon(bytes)`, `tray.setMenu([{item:{label,id,enabled}}])`, `menuclick` with `e.detail.id`, `click` event, and the `desktop.app.{name,identifier,icons}` / `desktop.backend` / `desktop.output` deno.json fields. No plan-file signature edits were required. One additional doc detail worth carrying into Task 2/3: the "Working directory & assets" page explicitly documents that compiled binaries run with cwd set to the *user's* cwd, not the bundle dir — reinforcing the plan's existing `Deno.chdir()`-based design.

### Task 3 findings (`src-deno/server.ts`, 2026-07-05)

- **cwd split does NOT work — symlink fallback required, confirmed empirically.** The brief's primary plan was: `Deno.chdir()` to the writable app-data dir before importing the standalone `server.js`, so Next reads/writes `data/` there while still serving `.next/static` relative to its own `__dirname`. This fails because **Next.js's generated standalone `server.js` itself calls `process.chdir(__dirname)` on line 6** (baked into the build output, not something we control), which runs *after* our chdir and silently resets cwd back to `src-deno/standalone/`. Proof: after our chdir + import, `lsof -p <pid> | grep cwd` showed the OS-level process cwd was still `.../src-deno/standalone`, and `POST /api/scan`'s `"synced"` event reported writing to `src-deno/standalone/data/projects_metadata.jsonl` instead of the app-data dir. Fix implemented (the brief's documented fallback): don't chdir at all; instead remove/replace `src-deno/standalone/data` with `Deno.symlink(appDataDir/data, standaloneDir/data, { type: "dir" })` before importing `server.js`. Verified: after the swap, `POST /api/scan` still reports `"synced"` against the path inside `standalone/data/`, but that path is now a symlink — confirmed same inode as the app-data-dir file via `stat -f "%i"` on both paths, and the app-data copy's mtime updated immediately after the scan (`00:08:52` → `00:14:47`). `.env.local` doesn't need the same treatment — we `Deno.env.set()` every key from it into the process env ourselves before Next starts, so it's populated regardless of which directory ends up being cwd.
- **New, previously-unobserved requirement: `deno run` needs `--unstable-detect-cjs`.** Running `deno run -A src-deno/server.ts` (no other flags) crashes immediately on `await import(".../standalone/server.js")`: `ReferenceError: require is not defined` — Deno's ES-module loader doesn't treat `server.js` as CommonJS because `src-deno/standalone/package.json` has no top-level `"type": "commonjs"` (only a nested `repository.type: "git"` field, which is not the same key). Deno's own error hint names the fix: pass `--unstable-detect-cjs` (confirmed: the exact same command with that flag added starts cleanly — `✓ Ready in 46ms`, `GET /` → 200). This flag was **not** present in the plan's `deno:run`/`deno:build`/verification commands and needs to be added wherever `src-deno/server.ts` (via `main.ts`) is launched — i.e. `deno desktop -A src-deno/main.ts` likely needs to become `deno desktop -A --unstable-detect-cjs src-deno/main.ts` (untested under `deno desktop` specifically — the auto-detect spike in Task 1 used `next start`'s ordinary ESM entry, not our standalone `server.js` import, so it never hit this path). **Flagging for Task 4/2 owners:** verify whether `deno desktop` accepts `--unstable-detect-cjs` the same way `deno run` does, and update `package.json`'s `deno:run`/`deno:build` scripts and `deno.json` accordingly if so.

### Task 4 findings (`src-deno/main.ts` + `src-deno/tray.ts`, 2026-07-05)

- **CJS-flag question resolved: `deno desktop` accepts `--unstable-detect-cjs` identically to `deno run`.** Confirmed via `deno desktop --help=unstable` (lists `--unstable-detect-cjs` under the same top-level unstable-flags group used by `deno run`/`deno compile`) and empirically: `npm run deno:run`/`deno:build` now pass `-A --unstable-detect-cjs` and both build+run cleanly, importing the standalone `server.js` without the `ReferenceError: require is not defined` crash Task 3 hit. `package.json`'s `deno:run` and `deno:build` scripts were updated accordingly (this was flagged as in-scope for Task 4/2 by Task 3's notes above).
- **Compiled `deno desktop -A src-deno/main.ts` (no `-o`) compiles the `.app` per `deno.json`'s `desktop.output` and exits without auto-launching it**, exactly as Task 1 found for auto-detect mode. The built bundle lands at `dist/Stow Dashboard Deno.app` (from `deno.json`); running `npm run deno:run` is therefore a *build* step, not a *launch* step, in this environment. To see the window/tray you must separately launch `dist/Stow Dashboard Deno.app/Contents/MacOS/laufey_webview` (the compiled binary name is always `laufey_webview` regardless of the configured app name). This is worth a DX callout: there is no single `npm run deno:run` that both builds and shows a window today; a follow-up could `open` the bundle after building if that workflow is wanted, but wasn't in this task's scope.
- **New, more serious bug found and fixed: Task 3's `server.ts` symlink-swap crashes under the *compiled* app with `NotSupported`.** Task 3 verified the symlink approach (`Deno.symlink(appDataDir/data, STANDALONE_DIR/data)`) only under `deno run`. Under the compiled `deno desktop` binary, `STANDALONE_DIR` resolves inside Deno's self-extracting VFS temp dir (`$TMPDIR/deno-compile-<binary>/src-deno/standalone/`), and that directory is **fully read-only** — not just for symlinks. Direct probe (`deno compile` a script that `Deno.mkdir`/`Deno.writeTextFile`/`Deno.remove`/`Deno.symlink`s inside its own `import.meta.url` dir) showed all four operations fail there, e.g. `mkdir FAILED: mkdir '.../deno-compile-probe/subdir'`. So no in-place write under `STANDALONE_DIR` is possible in a compiled app, ever — this invalidates any strategy that touches files under the app's own module directory at runtime.
  - **Fix implemented:** don't touch `STANDALONE_DIR` at all. Instead, `Deno.chdir(appDataDir)` right before importing `server.js`, then monkeypatch `process.chdir = () => {}` (a no-op) so Next's own baked-in `process.chdir(__dirname)` (in the generated `server.js`) can't reset it back. This works because Next's asset serving (`.next/static`, etc.) uses the `dir` value passed into `startServer({ dir, ... })`, not `process.cwd()` — only app-level code (`src/lib/projects.js`, and `src/app/api/*/route.js`'s module-level `path.join(process.cwd(), ...)` constants like `SYNC_FILE`/`DATA_FILE`/`ENV_PATH`) reads cwd, and chdir'ing before Next preloads those route modules (Next 16's `experimental.preloadEntriesOnStart: true` loads them during the same startup window, not lazily per-request as initially assumed — a chdir *after* `await import("./server.js")` resolves is too late for those modules, verified by the `"synced"` event still reporting the old `standalone/data/...` path when tried that way first) makes them observe the correct writable dir. Verified end-to-end under `deno run -A --unstable-detect-cjs src-deno/server.ts`: `GET /` 200 with real table, `GET /api/settings` reads app-data `.env.local`, `POST /api/scan` reports `"synced"` against `~/Library/Application Support/StowDashboardDeno/data/projects_metadata.jsonl` with mtime updated. Re-verified under the compiled app (see below).
- **Third bug found and fixed: `deno desktop` ignores the requested `PORT` and unconditionally rebinds to its own auto-assigned port, exposed via `DENO_SERVE_ADDRESS`.** Confirmed by diagnostic instrumentation in both `server.ts` and a patched copy of `server.js`: `process.env.PORT` correctly read `"3087"` at every observation point (right before import, and inside `server.js` itself, `currentPort=3087`), Next's `startServer({port: 3087, ...})` was called with the right value, `server.listen(3087, 'localhost')` was invoked — and yet the actual bound TCP port was different every run (e.g. `56506`, `56787`, `57104`...) and **exactly matched** `DENO_SERVE_ADDRESS` (e.g. `tcp:127.0.0.1:56506`) each time. `http://localhost:3087` was unreachable (`curl` connection refused) while the `DENO_SERVE_ADDRESS` port served the real app. Deno's own docs (`docs.deno.com/runtime/desktop/serving/`) confirm this is intentional for `Deno.serve()` — "You cannot override the port `Deno.serve()` binds to inside `deno desktop`... the webview needs to navigate to the same port the runtime is listening on, and the runtime is the source of truth for that value" — and our testing shows the interception is transitive to Node-compat's `http.Server.listen()` (which Next's standalone `server.js` uses) too; the docs don't call this out explicitly for Node-compat, so this is a documentation gap we're closing here.
  - **Fix implemented:** `server.ts` now exports `getActualBaseUrl()`, which reads `DENO_SERVE_ADDRESS` (format `tcp:<host>:<port>`) when present and derives the real `http://` URL from it, falling back to the fixed `BASE_URL`/`PORT=3087` constants when the env var is absent (i.e. under plain `deno run`, where we verified `DENO_SERVE_ADDRESS` is unset and our requested port 3087 binds exactly as asked). `main.ts` calls `getActualBaseUrl()` after `startServer()` and uses that — not the static `BASE_URL` export — for `waitForServer()`, `win.navigate()`, and the tray's rescan `fetch()` base. Verified: compiled app's window loaded the real dashboard (title "Stow Dashboard", 1115-project table) at the `DENO_SERVE_ADDRESS`-derived URL; the old hardcoded-3087 approach would have pointed the window at a dead port.
- **Window verified visually.** `screencapture` + accessibility (`System Events` window list for process `laufey_webview`) confirmed a real window titled exactly "Stow Dashboard (Deno)", positioned/sized `1400×900` as requested, rendering the live dashboard (real scan data, "Synced Ns ago", full project table) — not a blank/error page.
- **Rescan handler verified functionally.** `curl -X POST http://<actual-host>:<actual-port>/api/scan` (the exact one-line fetch the tray's "Rescan Projects" `menuclick` handler performs) returns the real SSE scan stream and the dashboard's "Synced" timestamp updates afterward — confirmed by re-screenshotting after a manual curl-triggered rescan.
- **Quit/server-death parity verified.** Killing the app's OS process (`kill <pid>`, simulating `Deno.exit(0)` from the tray's Quit handler or Cmd+Q) reliably takes the in-process Next server down with it — `lsof -nP -iTCP:<port> -sTCP:LISTEN` was empty immediately after, every time, matching Tauri's in-process-child-death behavior with no orphan-process risk.
- **Tray icon/menu interaction: found a reproducible crash, NOT verified as working.** `Deno.Tray()`'s status-bar item is confirmed to exist — `System Events` reports exactly 1 `menu bar item` for the `laufey_webview` process with `help` text `"Stow Dashboard (Deno)"` (matching `tray.setTooltip(...)`), i.e. the tray object was constructed successfully and the app didn't crash on startup. However, **every attempt to interact with it via macOS Accessibility (`click itm`, and separately `perform action "AXPress" of itm`) caused the app process to disappear silently within ~1s** — no stderr, no exception, no macOS crash report in `~/Library/Logs/DiagnosticReports`, no system log entry (`log show --predicate 'processImagePath CONTAINS "laufey_webview"'` was empty around the crash time) — the process and its listening port both simply vanished. This reproduced 2/2 times (`click`, then separately `AXPress`), and did NOT happen at all across a 30s idle window with zero interaction, isolating the crash to the interaction itself rather than a timeout/idle bug. The tray icon is also not visually distinguishable in menu-bar screenshots at 2x zoom (no new icon appeared versus a pre-launch baseline capture), despite Accessibility confirming the status item exists — possibly the 22×22 PNG isn't rendering, or it's a known rough edge of this experimental API surface. **This could be an Accessibility-driven-synthetic-click artifact specific to this experimental API rather than a real end-user bug** (a physical mouse click was not tested — no `cliclick` available in this environment, and computer-use/GUI-automation tools were out of scope for this non-interactive verification pass) — but as shipped, this is unverified and risky. Flagging as a human-required smoke test (see report) rather than claiming it works.
- **File-size bloat, pre-existing, not introduced by this task:** repeated `npm run deno:build` runs without first `rm -rf dist/` accumulate — one observed run produced a 1.99GB `.app` (vs. ~458MB on a clean build), apparently because the compiler's dependency/file dedup doesn't fully collapse across incremental builds when the previous `.app` (itself embeddable-looking under `dist/`) is left in place. Not investigated further (out of scope for window/tray parity), but worth a `rm -rf dist/` reminder in the build docs, and/or investigating in Task 6 alongside the from-scratch size comparison.
